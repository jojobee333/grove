# DNS Architecture and How Attackers Abuse It

**Module**: M02 · DNS — Single Protocol, Every Domain
**Type**: core
**Estimated time**: 13 minutes
**Claim**: C2 from Strata synthesis

---

## The core idea

DNS is the infrastructure beneath every other protocol. Before any HTTP request can be sent, any TLS handshake initiated, or any authentication event occur, a DNS lookup resolves the target name to an IP address. This architectural position makes DNS uniquely valuable to attackers: compromise or abuse DNS and you gain leverage across every protocol simultaneously.

Attackers use DNS for three distinct purposes. As an **attack target**: poison a recursive resolver's cache and every client it serves is silently redirected to attacker-controlled infrastructure. As **C2 infrastructure**: encode commands and data in DNS queries and responses to reach compromised machines through firewalls that permit port 53 outbound. As **reconnaissance**: zone transfers from misconfigured authoritative servers hand an attacker a complete map of an organization's internal topology.

Defenders use the same DNS infrastructure for certificate authority restriction (CAA records), process-level query monitoring (Sysmon Event 22), and topology protection (split-horizon DNS). The same protocol that attackers love is the protocol defenders instrument. This is C2 from Claim 2: DNS is a universal attack, defense, and monitoring surface across all four security domains [S005](../../research/security-foundations/01-sources/web/S005-rfc1034-dns.md) [S006](../../research/security-foundations/01-sources/web/S006-mitre-attack-c2.md) [S013](../../research/security-foundations/01-sources/web/S013-owasp-tls-cheatsheet.md) [S015](../../research/security-foundations/01-sources/web/S015-sysmon-v15.md).

## Why it matters

In incident investigations, DNS query logs are among the first telemetry sources examined — and often the most revealing. DNS C2 is the dominant command-and-control technique catalogued in MITRE ATT&CK TA0011 because UDP port 53 is permitted outbound through nearly every corporate firewall, and DNS traffic blends into the enormous volume of legitimate lookups generated by any active endpoint. An analyst who is not looking at DNS during an investigation is ignoring the most likely active C2 channel [S006](../../research/security-foundations/01-sources/web/S006-mitre-attack-c2.md).

Understanding the DNS trust chain also explains why cache poisoning is possible at all, and why DNSSEC was retrofitted rather than built-in: the original DNS design trusted the network to be cooperative, just as TCP and ARP did.

## A concrete example

**Example 1 — Kaminsky-style cache poisoning**

DNS uses a recursive resolver: when your laptop needs to reach `api.example.com`, it sends a query to a configured resolver (often your ISP or a corporate DNS server). That resolver queries the authoritative name servers and caches the result for the duration of the record's TTL.

The trust assumption: the resolver will receive the authoritative answer before a forged response arrives, and will accept only the answer matching the 16-bit transaction ID.

The Kaminsky attack (2008) exploits this race condition. An attacker who knows a resolver is about to query for a domain (by triggering the lookup themselves, or by targeting a predictable query) floods the resolver with forged DNS responses, each containing a different transaction ID. With UDP port randomisation disabled or guessable, hitting the correct ID requires only around 65,536 attempts per query — feasible in milliseconds. A successful hit poisons the resolver's cache: every client it serves now receives the attacker's IP for the target domain, for the full TTL. A single poisoned cache entry can redirect thousands of users [S005](../../research/security-foundations/01-sources/web/S005-rfc1034-dns.md).

**Example 2 — DNS-based command and control (tunneling and DGA)**

An implant on a compromised host encodes data in DNS query subdomains: `Y2hlY2tpbi5leGFtcGxl.c2domain.attacker.com`. The attacker controls the authoritative name server for `c2domain.attacker.com`. The DNS resolver forwards the query; the attacker's server decodes the data from the query labels and returns commands encoded in TXT or A records. The implant decodes the response. No direct TCP connection to the attacker is required — DNS carries the entire C2 channel [S006](../../research/security-foundations/01-sources/web/S006-mitre-attack-c2.md).

Domain Generation Algorithms (DGA) extend this resilience. The implant computes hundreds of pseudo-random domain names daily from a shared seed. The attacker registers only a few. The malware tries them all until one resolves. Blocking individual C2 domains is useless — there are thousands of potential names. Detection requires identifying the pattern of failed NXDOMAIN lookups rather than any specific domain name (MITRE T1568.002).

**Example 3 — Zone transfer exposure**

DNS zone transfers (AXFR queries, TCP port 53) are designed to let secondary name servers replicate zone data from primaries. A misconfigured zone that permits AXFR from any IP gives any attacker who sends `dig AXFR @ns1.example.com example.com` a complete exported list of every hostname and IP address in the zone — a free reconnaissance map of the organization's internal infrastructure [S005](../../research/security-foundations/01-sources/web/S005-rfc1034-dns.md).

## Limitations

Monitoring DNS for C2 requires investment in two data sources: Sysmon Event 22 on endpoints (capturing which process made each query) and DNS server query logs (capturing query/response pairs at the network level). Coverage is incomplete without both — endpoint DNS logs without server logs miss lateral movement to internal hosts; server logs without process attribution cannot identify which compromised process is running the C2 beacon. Neither data source alone is sufficient [S015](../../research/security-foundations/01-sources/web/S015-sysmon-v15.md).

## Key points

- DNS cache poisoning exploits the TTL/transaction-ID race condition in recursive resolution — one successful attack redirects all clients of a poisoned resolver
- DNS tunneling and DGA are the dominant C2 techniques in real intrusions because port 53 traverses most firewalls and blends with legitimate traffic
- Zone transfer exposure is a misconfiguration, not a protocol vulnerability — it requires no attack; AXFR just needs to be permitted
- Effective DNS C2 detection requires both endpoint (process-level) and network (query/response) logging simultaneously
