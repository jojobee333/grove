# L27 — From Gap Report to Production-Ready Implementation

**Module**: M10 — Code Efficacy Assessment  
**Type**: Applied  
**Estimated time**: 30 minutes

---

## The Goal of This Lesson

You have built Meridian. You have the Five-Question Framework. You know what the common gaps look like. This lesson shows you how to use them together — systematically auditing your own implementation and producing a structured remediation plan that moves you from "working code" to "production-ready code."

---

## Step 1: Run the Five-Question Framework as a Code Audit

For each of the five questions, perform a targeted search across your codebase.

### Audit Script (run from the repo root)

```bash
echo "=== Q1: Queries missing workspace_id ==="
# Look for .where() calls without workspace_id — review each result manually
grep -rn "\.where(eq(" server/src --include="*.ts" | grep -v "workspace_id"

echo ""
echo "=== Q2: Routes missing safeParse ==="
# Look for route handlers without safeParse in the first block
grep -n "router\.\(post\|put\|patch\)" server/src/routes --include="*.ts" -l

echo ""
echo "=== Q3: AI provider calls missing audit log ==="
grep -n "provider\.\(complete\|stream\)" server/src --include="*.ts" -B5 | grep -v "logAuditEvent"

echo ""
echo "=== Q4: bcrypt cost not equal to 12 ==="
grep -rn "bcrypt\.hash(" server/src --include="*.ts"

echo ""
echo "=== Q5: Refresh handler — check for reuse detection ==="
grep -n "rt:" server/src/routes/auth.ts
```

These are grep heuristics — manual review is required for each result. The grep output tells you where to look, not what the verdict is.

## Step 2: Produce a Gap Report

For each violation found, write a structured gap entry:

```markdown
## Gap Report — [Date]
Generated by: Five-Question Efficacy Framework

### GAP-001
- **Question**: Q1 — workspace_id in WHERE
- **File**: server/src/routes/pages.ts
- **Line**: 87
- **Description**: `db.select().from(blocks).where(eq(blocks.page_id, pageId))` omits workspace_id
- **Risk**: Cross-tenant block data leakage if RLS context not set
- **Fix**: Add `eq(blocks.workspace_id, req.workspace.id)` to the WHERE clause

### GAP-002
- **Question**: Q3 — Audit log before AI call
- **File**: server/src/routes/ai.ts
- **Line**: 134
- **Description**: `provider.stream(...)` called before `logAuditEvent`
- **Risk**: AI stream actions not logged if provider throws on initialization
- **Fix**: Move `logAuditEvent` call to before `provider.stream(...)`
```

---

## Step 3: Prioritize by Risk Level

Not all gaps are equal. Prioritize by the potential impact if exploited:

| Priority | Gap types |
|---|---|
| P0 — Fix immediately | Missing `workspace_id` (cross-tenant leakage), `SET` not `SET LOCAL` (context bleed) |
| P1 — Fix before first user | Missing audit log, bcrypt cost < 12 |
| P2 — Fix before public launch | Missing Zod validation, user input in system prompt |

P0 gaps must be fixed before the next commit. P1 gaps before any user data enters the system. P2 gaps before external traffic.

---

## Step 4: Apply Fixes in Isolation

Fix one gap at a time. Each fix should be a separate, focused commit:

```
[security] Add workspace_id to blocks query in pages route

Fixes GAP-001. The WHERE clause in GET /pages/:id/blocks was missing
workspace_id, relying solely on RLS for tenant isolation. Explicit
workspace_id filter added per architecture non-negotiable rules.

Closes: GAP-001
```

Focused commits make gap fixes easy to review, revert if needed, and reference in the gap log.

---

## Step 5: Re-run the Framework After Each Fix

After applying a fix, run the same grep query from Step 1 for that specific question. Verify the violation no longer appears. Do not mark a gap as resolved until the re-run confirms it.

## Ongoing: The CI Efficacy Check

After your initial audit is clean, add the framework checks to CI:

```yaml
# .github/workflows/efficacy.yml
name: Efficacy Check

on: [push, pull_request]

jobs:
  five-question-check:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      
      - name: Q4 — Check bcrypt cost factor
        run: |
          # Fail if any bcrypt.hash() call does not use 12
          if grep -rn "bcrypt\.hash(" server/src --include="*.ts" | grep -v ", 12)"; then
            echo "❌ bcrypt cost factor is not 12"
            exit 1
          fi
          echo "✅ bcrypt cost factor is 12 everywhere"
      
      - name: Q1 — Check for workspace_id in queries (advisory)
        run: |
          echo "Advisory: Review these WHERE clauses for missing workspace_id:"
          grep -rn "\.where(eq(" server/src --include="*.ts" | grep -v "workspace_id" || true
```

The Q4 check can be fully automated (exact grep pattern). The Q1 check is advisory — grep cannot prove the presence of workspace_id in every Drizzle query pattern, but it flags suspicious results for manual review.

## What "Production-Ready" Means in Meridian

A Meridian implementation is production-ready when:

1. All five framework questions return zero violations
2. The audit_log table has an entry for every AI action in the test suite
3. The adversarial RLS test (from L22) passes
4. All bcrypt hashes in the codebase use cost 12
5. The refresh token handler has a reuse detection branch
6. No user-provided text appears in any system prompt string

This is not comprehensive security — it is the minimum bar defined by the architecture's non-negotiable rules. Production security also requires dependency audits, penetration testing, infrastructure hardening, and ongoing monitoring.

---

## Key Points

- The gap audit is systematic, not intuitive — use grep patterns and review every result.
- Gaps are prioritized P0/P1/P2 by blast radius: tenant leakage first, audit/crypto second, input validation third.
- Fix one gap per commit. Small, reversible, reviewable.
- Re-run the specific framework question after each fix to confirm resolution.
- CI efficacy checks automate the verifiable questions (bcrypt cost) and surface advisory results for manual review.
- "Production-ready" is a checklist, not a feeling — every item on the checklist has a verifiable answer.
